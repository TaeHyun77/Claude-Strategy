### 서브에이전트 디스패칭

---

**개념 정리**

메인 에이전트가 작업을 수행하는 동안, 별도의 서브에이전트를 백지 컨텍스트로 띄워서 같은 문제를 독립적으로 풀게 한 뒤 결과를 비교하는 방식입니다. 메인 에이전트는 이미 대화 맥락에 영향을 받고 있으므로, 편향 없는 서브에이전트의 판단과 교차 검증하면 타당성 검증이 가능합니다.

**왜 효과적인가**

메인 에이전트는 대화가 쌓이면서 "이전에 이렇게 하기로 했으니까"라는 **확증 편향(Confirmation Bias)**에 빠질 수 있습니다. 예를 들어 초반에 잘못된 아키텍처 결정을 했는데, 컨텍스트에 그 결정이 남아있으면 계속 그 방향으로 갑니다. 백지 상태의 서브에이전트는 이런 편향 없이 순수하게 "이 코드, 이 구조가 맞나?"를 판단할 수 있습니다.

**Claude Code에서의 실현 방법**

Claude Code는 실제로 `--print` 모드나 스크립트를 통해 서브프로세스를 띄울 수 있습니다:

```kotlin
# 메인 에이전트가 작업한 결과물을 서브에이전트에게 검증 요청
claude --print "이 코드를 리뷰해줘: $(cat src/Service.java)" > review.md
```

또는 Task 도구를 활용해서 서브에이전트를 생성하는 것도 가능합니다.

**개선 및 의견**

이 방식의 핵심 가치는 `독립적 시각의 확보`인데, 몇 가지 고려할 점이 있습니다.

서브에이전트가 완전히 백지면 프로젝트 맥락을 모릅니다. 따라서 완전 백지보다는 프로젝트 기본 컨텍스트만 주입하고 대화 히스토리는 제외하는 게 더 효과적일 것 같습니다.

비교 전략도 정해야 합니다. 메인과 서브의 결과가 다를 때 누구 말을 들을지, 단순히 사람이 판단할지, 아니면 **제3의 에이전트가 중재**할지 등의 워크플로우를 설계해야 합니다.

실용적으로 추천하는 구조는 이렇습니다:

```kotlin
메인 에이전트: 작업 수행 → 결과물 생성

서브 에이전트 A: 백지 + CLAUDE.md만 → 같은 요구사항으로 독립 작업

서브 에이전트 B: 백지 + CLAUDE.md만 → 두 결과물 비교 리뷰

→ 사람이 최종 판단
```

### Compaction (컨텍스트 자동 요약) 대응 전략

---

**개념 정리**

Claude Code는 대화가 길어져 컨텍스트 윈도우가 가득 차면 자동으로 이전 대화를 요약( Compaction )합니다. 이 과정에서 세부 정보가 손실되어 Claude가 속히 멍청해지는 현상이 발생할 수 있습니다. 

**실제로 어떤 정보가 손실되나**

요약 과정에서 가장 먼저 사라지는 것들은 다음과 같습니다:

- 구체적인 코드 변경 내용 ( 어떤 줄을 왜 바꿨는지 )
- 시도했다가 실패한 접근 방식들
- 중간에 논의했던 트레이드오프와 결정 이유
- 에러 메시지의 정확한 내용

**대응 전략**

1. CLAUDE.md에 중간 상태를 기록하기
    
    이것이 가장 실용적입니다. CLAUDE.md는 항상 컨텍스트에 포함되므로 Compaction의 영향을 받지 않습니다:
    
    ```kotlin
    # 현재 작업 상태
    ## 진행중: 채팅 시스템 리팩토링
    - [완료] ChatService에서 Kafka Producer 분리
    - [완료] WebSocket 핸들러 인증 로직 추가
    - [진행중] 메시지 읽음 처리 최적화
      - 배치 업데이트 방식 채택 (개별 UPDATE → IN 절 배치)
      - PlaneSubscription 쿼리 N+1 해결 필요
    - [예정] 테스트 코드 작성
    ```
    

1. 작업 단위를 작게 나누기
    
    Compaction이 발생할 정도로 대화가 길어지기 전에 작업을 끝내는 겁니다. 
    

1. 주기적으로 명시적 요약 요청
    
    Compaction이 자동으로 되기 전에 직접 요약을 요청합니다
    
    ```kotlin
    지금까지 작업 내용을 xxx.md에 정리해줘. 변경한 파일, 변경 이유, 남은 작업을 포함해서.
    ```
    

**의견**

가장 현실적인 조합은 전략 1 + 전략 2입니다. xxx.md를 작업 일지처럼 활용하면서, 하나의 세션에서 하나의 명확한 작업만 수행하는 겁니다. 장시간 복잡한 작업을 한 세션에서 하려는 것 자체가 LLM 에이전트와의 협업에서는 안티패턴이라고 볼 수 있습니다.

### PR 코멘트 자동 분석 및 대응

---

**개념 정리**

내가 올린 PR에 리뷰어가 코멘트를 달면, AI가 해당 코멘트의 의도를 분석하고 몇 가지 대응 방안을 제시 → 개발자가 검토 후 선택하여 반영하는 자동화입니다.

**구체적인 워크플로우**

```kotlin
1. 본인이 PR 생성

2. 리뷰어 : PR에 대한 의견 제시
           
3. AI 분석 : 코멘트 의도 파악
       + 해당 코드 확인
       + 리뷰어의 의견 검증 및 타당성 파악 + 여러 의견 제시
           
4. 개발자 : AI의 분석 자료를 살펴보고 의사 결정
           
5. AI : 리뷰 코멘트에 답변 작성 + 필요시 코드 수정 커밋
```

**실현 방법**

GitHub MCP 서버가 이미 존재하므로, 이를 활용할 수 있을 것 같습니다.

```kotlin
// 필요한 MCP Tool
{
  "tools": [
    "get_pr_comments",      // PR의 리뷰 코멘트 조회
    "get_pr_diff",          // PR의 변경 내용 조회  
    "reply_to_comment",     // 코멘트에 답변
    "push_commit"           // 수정 사항 커밋
  ]
}
```

GitHub Actions나 Webhook으로 트리거를 걸 수도 있습니다:

```kotlin
# .github/workflows/pr-review-assist.yml
on:
  pull_request_review_comment:
    types: [created]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - name: AI 분석 및 대응안 생성
        run: |
          claude --print "이 PR 코멘트를 분석하고 대응안을 제시해줘:
          코멘트: ${{ github.event.comment.body }}
          파일: ${{ github.event.comment.path }}
          라인: ${{ github.event.comment.line }}"
```

**개선 및 의견**

이 아이디어는 좋지만, 자동화 범위를 신중하게 설정해야 합니다.

자동화하기 좋은 코멘트 유형은 오타 지적, 네이밍 컨벤션, 미사용 import, 간단한 코드 스타일 등입니다. 이런 건 AI가 바로 수정하고 커밋해도 됩니다.

사람 판단이 반드시 필요한 유형은 아키텍처 결정, 비즈니스 로직 변경 제안, 성능 트레이드오프 논의 등입니다. 이런 건 대응안만 제시하고 개발자가 선택해야 합니다.

위험한 부분은 AI가 리뷰어의 의도를 잘못 파악하는 경우입니다. "이 부분 좀 이상한데요?"라는 모호한 코멘트를 AI가 자의적으로 해석해서 엉뚱한 대응안을 내놓을 수 있습니다. 따라서 AI가 직접 답변을 게시하는 것은 피하고, 항상 개발자에게 초안을 보여주는 단계를 두는 게 안전합니다.

### 실시간 에러 감지 → MCP → AI 분석 파이프라인

---

**개념 정리**

운영 또는 개발 환경에서 에러가 발생하는 즉시 MCP를 통해 스택 트레이스를 실시간으로 수집하고, AI가 현재 코드베이스와 대조하여 원인 분석 + 해결 방안을 제시 → 개발자가 선택하여 적용하는 워크플로우입니다. 

핵심은 발생 즉시 알림 ( slack 등등.. ) → 즉시 분석이라는 실시간성에 있으며, 최근 N시간 조회는 보조 기능으로 사용할 수 있을 것 같습니다.

**실시간 파이프라인 아키텍처**

```kotlin
[운영/개발 서버]
	예외 발생 즉시

[MCP 서버 - 실시간 수신]
  스택 트레이스 수신
  민감 정보 마스킹
  중복 에러 필터링
  
       ↓ (실시간 Push)
       
[Claude Code / 개발자 터미널]
  즉시 알림 + 코드 분석
  해결안 제시
  
       ↓
       
  개발자 선택 → 코드 수정 → 커밋
```

**MCP 서버 설계**

실시간이 주이므로 **Push 기반 설계**가 핵심입니다:

```kotlin
{
  "tools": [
    {
      "name": "subscribe_errors",
      "description": "실시간 에러 스트림 구독. 에러 발생 즉시 Push",
      "parameters": {
        "severity": ["ERROR", "FATAL"],
        "services": ["chat-back", "api-gateway"]
      }
    },
    {
      "name": "get_error_context",
      "description": "발생한 에러의 전후 로그, 요청 파라미터, 사용자 정보 조회",
      "parameters": { "errorId": "string", "logLines": 50 }
    },
    {
      "name": "get_source_code",
      "description": "스택 트레이스에서 참조된 소스 파일의 해당 라인 조회",
      "parameters": { "className": "string", "lineNumber": "int" }
    }
  ],
  "notifications": [
    {
      "name": "error_occurred",
      "description": "에러 발생 시 즉시 전달되는 알림",
      "payload": {
        "errorId": "string",
        "timestamp": "ISO8601",
        "exceptionClass": "string",
        "message": "string",
        "stackTrace": "string",
        "service": "string",
        "endpoint": "string"
      }
    }
  ]
}
```

**실시간 워크플로우 시나리오**

```kotlin
[14:32:05] 운영 서버에서 NullPointerException 발생
                ↓ 즉시
[14:32:06] MCP 서버가 에러 수신 → Claude에게 Push

Claude: "⚠️ 에러가 발생했습니다.
         
         서비스: xxx
         시간: xxx
         위치: xxx
         예외: NullPointerException
         요청: POST /api/chat/send (userId=1023)
         
         [코드 분석 중...]
         
         원인: xxx
         
         해결안 1: aaa
         해결안 2: bbb
         해결안 3: ccc
         
개발자: "x번으로 해줘"

Claude: [코드 수정 + 테스트 작성 + 커밋]
```

**개발 환경에서의 활용**

운영뿐 아니라 로컬 개발 중에도 동일한 파이프라인을 적용할 수 있습니다:

이렇게 되면 개발자가 콘솔에서 스택 트레이스를 직접 읽고 구글링하는 과정이 에러 발생 → AI가 바로 원인 파악 → 해결안 선택으로 단축됩니다.

**실시간 처리 시 고려사항**

에러 폭주 대응 : 장애 상황에서는 같은 에러가 초당 수백 건 발생할 수 있습니다. MCP 서버 단에서 디바운싱과 집계를 처리해야 합니다. 전부 Claude에게 넘기면 컨텍스트가 많이 소진됩니다.

우선순위 자동 판단도 중요합니다. 모든 에러를 즉시 알리면 알림 피로가 생깁니다. MCP 서버에서 심각도를 판단하는 기준이 필요합니다. WARN 이하는 요청 시에만 조회하는 식의 계층 구조가 현실적일 것 같습니다.

컨텍스트 보존 전략도 필요합니다. 에러 하나 분석할 때마다 스택 트레이스 + 소스 코드 + 로그가 컨텍스트에 쌓이므로, 에러 분석 세션은 "하나의 에러 해결 → `/clear` → 다음 에러" 식으로 세션을 분리하거나, 2번에서 논의한 CLAUDE.md 기록 전략과 병행해야 합니다.

보안 측면에서는 스택 트레이스에 포함될 수 있는 DB 비밀번호, JWT 토큰, 사용자 개인정보 등을 MCP 서버 단에서 마스킹 처리 후 전달해야 합니다.
