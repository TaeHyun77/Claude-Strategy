# Claude-Strategy
Cladue 활용법 정리<br><br><br>

### Claude Code 프로젝트 설정 가이드

---

Claude Code를 프로젝트에 도입할 때 사용하는 나만의 설정 구조와 활용법을 정리한 문서입니다.<br><br>

**설정 파일 구조**

```kotlin
project-root/
├── CLAUDE.md 프로젝트 컨텍스트 (진입점)
│
├── rules/ 상세 규칙 문서
│ ├── architecture.md 아키텍처 / 디렉터리 구조 / 데이터 흐름
│ ├── pattern.md 코딩 패턴
│ ├── errorHandler.md 에러 핸들링 규칙
│ └── commit.md 커밋 컨벤션
│
└── .claude/
	└── commands/ 커스텀 슬래시 명령어
		└── pr-git.md /pr-git 명령어 정의
```
<br>

### 각 파일의 역할

---

**CLAUDE.md — 프로젝트 컨텍스트 진입점**

Claude Code가 대화 시작 시 자동으로 읽는 파일입니다. 프로젝트의 전체 맥락을 빠르게 파악할 수 있도록 요약 수준으로 작성하되, 구체적인 내용은 임포트하는 방식으로 사용합니다.<br><br>

[ 포함하는 내용 ]

- 프로젝트 설명: 한 줄 요약 ( 기술 스택 + 도메 인)
- 응답 언어 지침: 한글/영문 사용 기준
- 기술 스택: 사용 중인 프레임워크, DB, 메시징 등
- 아키텍처: 주요 모듈 목록 + 상세 문서 참조
- 커밋 규칙: 핵심 규칙 요약 + 상세 문서 참조
- 패턴: 코딩 패턴 요약 + 상세 문서 참조
- 에러 핸들링: 예외 처리 방식 요약 + 상세 문서 참조
- 문서 동기화 규칙: 코드 변경 후 문서 업데이트 강제<br><br>

[ 작성 원칙 ]

- 각 섹션은 핵심만 작성하고 상세 내용은 @rules/파일명.md로 분리
- CLAUDE.md는 최대한 짧게 유지
- 필요 시 Claude가 rules 파일을 추가로 읽도록 유도<br><br>

**rules - 상세 규칙 문서**

CLAUDE.md에서 참조하는 상세 규칙 파일들을 모아두는 디렉터리입니다. 각 파일은 하나의 주제만 깊게 다룹니다.<br><br>

- rules/architecture.md
    
    프로젝트의 물리적 구조와 데이터 흐름을 정의하며, 기능 구현 시 파일 위치, 구조 설계 판단 기준으로 사용합니다.
    
    - 전체 디렉터리 구조
    - 파일 역할 설명
    - 데이터 흐름
    - 도메인 구조 패턴<br><br>
    
- rules/pattern.md
    
    코드 작성 시 따라야 하는 패턴과 규칙을 정의합니다.
    
    - DTO 패턴
    - 엔티티 구조
    - 트랜잭션 처리 방식
    - 네이밍 규칙<br><br>
    
- rules/errorHandler.md
    
    예외 처리 기준을 정의합니다.
    
    - 예외 클래스 구조
    - 계층별 역할
    - 에러 처리 방식
    - HttpStatus 기준<br><br>
    
- rules/commit.md
    
    커밋 메시지와 브랜치 전략을 정의합니다.
    
    - 커밋 메시지 규칙
    - 타입 정의 ( feat, fix, refactor 등 )
    - 브랜치 네이밍
    - 커밋 단위 기준<br><br>

**.claude/commands — 커스텀 명령어**

반복 작업을 슬래시 명령어로 정의하여 자동화합니다.<br><br>

[ 명명 규칙 ]

- 파일명이 명령어 이름이 됩니다.
- pr-git.md → /pr-git<br><br>

[ 작성 구조 ]

1. 명령어 설명
2. 참조 문서가 있다면 임포트
3. 플로우 설명 ( 단계별 )<br><br>

**문서 동기화 규칙**

코드 변경 시 관련 문서도 반드시 함께 수정해야 합니다.

이 규칙을 CLAUDE.md에 명시하면, Claude가 코드 수정 후 문서 업데이트 여부를 반드시 확인하게 됩니다.<br><br>

**설정 설계 원칙**

1. 계층 분리
    
    CLAUDE.md는 요약, rules는 상세로 분리하여 컨텍스트 효율을 높입니다.
    
2. 하나의 파일 = 하나의 주제
    
    각 규칙 파일은 하나의 역할만 가지도록 설계합니다.
    
3. 명령어는 플로우 중심
    
    명령어는 단계별 실행 흐름으로 작성합니다.
    
4. 사용자 확인 필수
    
    커밋, 푸시, PR 생성 등은 반드시 사용자 확인 후 진행합니다.
    
5. 규칙은 코드와 함께 관리
    
    CLAUDE.md, rules, commands 모두 저장소에 포함시켜 환경이 바뀌어도 동일한 규칙을 유지합니다.
