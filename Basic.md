### **컨텍스트 엔지니어링**

---

> AI에게 작업을 요청할 때, 불필요한 정보는 줄이고 꼭 필요한 정보만 정확하게 전달하는 기술
> 

AI는 입력된 맥락을 바탕으로 요청을 해석하고 결과를 생성하기 때문에, 어떤 정보를 얼마나, 어떤 구조로 제공하느냐에 따라 결과 품질이 크게 달라집니다.

Claude Code 환경에서는 이러한 컨텍스트를 체계적으로 제공하기 위해 claude.md 파일을 활용합니다.

### Claude.md

---

> 
> 
> 
> Claude Code가 매 세션 시작 시 자동으로 읽는 마크다운 파일
> 

Claude Code는 세션 간 상태를 보존하지 않으므로, 이 파일이 사실상 유일한 `장기 기억` 역할을 합니다.

포함하는 내용 : 프로젝트 개요, 기술 스택, 폴더 구조, 코딩 스타일, 아키텍처 규칙, 개발 지침 등

Anthropic 공식 문서의 설명에 따르면, claude.md는 단순 참고 메모가 아니라 claude의 프롬프트 일부로 작동하는 고영향 문서라고 합니다.

**왜 CLAUDE.md가 필요한가**

LLM ( Large Language Model, 대규모 언어 모델 )은 자연어를 이해하고 생성할 수 있지만, 한 번에 처리할 수 있는 입력과 출력의 길이에는 제한이 있으며, 이를 `컨텍스트 윈도우`라고 합니다.

또한, LLM은 세션 간 상태를 자동으로 저장하지 않기 때문에, 새로운 세션이 시작되면 프로젝트에 대해 아무것도 모르는 상태에서 시작됩니다.

대화가 길어질수록 오래된 맥락은 점차 밀려나고 잊히게 되므로, 초기에 잘 이해했던 프로젝트 구조나 아키텍처 원칙도 여러 번의 질의응답을 거치면서 놓칠 수 있으며, CLAUDE.md는 세션마다 프로젝트의 핵심 정보와 규칙을 다시 불러오는 기준점 역할을 합니다.

**CLAUDE.md가 제공하는 효과**

1. 새로운 세션에서도 Claude가 프로젝트 구조와 기술 스택, 도메인 개념을 빠르게 파악할 수 있습니다.
2. 코드 스타일과 아키텍처 규칙이 일관되게 유지됩니다.
3. 긴 대화 중 일부 컨텍스트가 유실되어도 기준점을 복원할 수 있습니다.
4. 매번 반복해서 설명해야 하는 내용을 줄여 작업 효율이 높힙니다.
5. 범용 AI를 프로젝트 맞춤형 개발 에이전트처럼 활용할 수 있습니다.
****

**Claude.md 관리 5원칙**

1. LLM은 약 150~200개의 지시사항을 안정적으로 따를 수 있으므로, 보편적이고 짧고 명확하게 300줄 이하를 유지하자
2. 추측 불가능한 정보만 넣기, 디렉터리 트리, API 엔드포인트, 설정 값 등을 작성하지 않습니다.
    
    ex ) 이 프로젝트만의 디렉터리 구조, 내부 도메인 규칙, 여러 규칙 등등
    
3. 세부 내용은 Progressive Disclosure로 분리하기
    
    모든 내용을 한 파일에 넣지 말고, 핵심만 남기고 세부 문서는 별도로 분리한 뒤 필요할 때 불러오는 구조를 사용하는 것이 좋습니다.
    
    ```jsx
    CLAUDE.md → 핵심 개요와 문서 안내
    agent_docs/building_the_project.md
    agent_docs/running_tests.md
    agent_docs/code_conventions.md
    ```
    
    대형 프로젝트에서는 `.claude/rules/` 디렉터리에 경로 스코프 규칙을 두면, 해당 경로의 파일을 수정할 때만 자동으로 로드됩니다.
    
4. 실수가 발생하면 즉시 반영해 반복 실수를 줄이자
    
    Claude가 잘못된 결과를 생성한 경우, 같은 실수를 반복하지 않도록 CLAUDE.md에 해당 instruction을 기록합니다.
    
5. 주기적으로 리뷰하고 불필요한 내용을 정리해 최신 상태를 유지하자
    
    `/init`으로 초안을 만들 수는 있지만, 자동 생성된 내용을 그대로 신뢰하지 말고 한 줄씩 검토하여 개선해야합니다.
    

⭐⭐⭐⭐

대부분의 세션은 Plan 모드에서 시작하는 것이 좋습니다.  ( Windows : `Shift+Tab` or `alt+m` )

Plan 모드에서 계획을 충분히 다듬은 뒤, Auto-accept 모드로 전환하여 실행하면 대체로 만족스러운 결과를 한 번에 얻을 수 있습니다.

1. 프로젝트의 WHAT(구조) , WHY(목적, 의도) , HOW(작동 방식) 을 간결히 담는 것이 좋습니다. ( 과하게 x )
2. LLM은 약 150~200개의 지시사항을 안정적으로 따를 수 있으므로 가능한 보편적이고 적은 지시사항을 추가하자
3. 대형 프로젝트에서는 모든 정보를 하나의 파일에 담기 어렵기 때문에, 필요한 정보만 단계적으로 제공하는 방식을 사용하는 것이 좋습니다.
    
    세부 지침은 agent_docs/ 폴더에 별도 마크다운 파일로 분리하고, CLAUDE.md에는 문서 목록, 간단한 설명, 필요 시 참조하라는 지시만 포함하며 코드 스니펫 대신 `file:line` 참조를 사용해 최신성을 유지하는 것이 좋습니다.
    
4. 스타일은 도구로 강제하고, AI는 사고와 설계에 집중하게 만들어라
5. /init 같은 기능으로 claude.md 초안을 만들 수는 있지만, 이 내용을 그대로 신뢰하고 사용하는 것은 추천되지 않습니다.
    
    최종 내용은 직접 검토하고 다듬는 것이 좋으며, 한 줄 한 줄이 실제로 필요한 규칙인지 확인하는 것이 좋습니다.
    

### Claude.md 구성 가이드

---

**포함시킬 것**

- 프로젝트 한 줄 소개
- 기술 스택
- 핵심 아키텍처 원칙
- 반드시 알아야 할 도메인 규칙
- 빌드/테스트/실행 명령어
- Gotchas ( Claude가 반복적으로 실수할 수 있는 함정 )
- 코딩 컨벤션  (이 프로젝트만의 비표준 패턴 )
- 관련 문서 목록 ( Progressive Disclosure 참조용 )

**포함시키지 말 것**

- 코드에서 직접 확인 가능한 정보 ( 패키지 트리, API 엔드포인트, 설정 값 )
- 과도한 스타일 규칙 ( 린터/포맷터로 자동화할 수 있는 것 )
- 일반적인 코딩 상식
- 현재 작업과 무관한 세부 절차
- 긴 코드 스니펫 ( 대신 `파일명:라인번호` 참조 )
- `/init`으로 자동 생성된 설명을 검토 없이 복붙한 내용

**WHY / WHAT / HOW 순서**

프로젝트의 목적(WHY) → 구조(WHAT) → 작동 방식(HOW) 순서로 구성하되, 과하지 않게 간결히 담아야합니다.

**강조는 아껴서**

`IMPORTANT`, `YOU MUST` 같은 강조 표현은 규칙 준수율을 높이지만, 진짜 중요한 항목에만 사용해야 효과가 있으며, 남발하면 모든 것이 강조되어 아무것도 강조되지 않습니다.

**작업 방식 권장 패턴 ⭐⭐⭐**

대부분의 세션은 Plan 모드( `Shift+Tab` 또는 `Alt+M` )에서 시작하는 것이 좋습니다.

계획을 충분히 다듬은 뒤, Auto-accept 모드로 전환하여 실행하면 대체로 만족스러운 결과를 한 번에 얻을 수 있습니다.

```jsx
### Plan Mode Default (CLAUDE.md 예시)
- 3단계 이상이거나 아키텍처 결정이 필요한 작업은 반드시 Plan 모드로 시작
- 문제가 발생하면 즉시 멈추고 재설계해야합니다.
- 검증 단계도 Plan 모드에서 설계합니다.
```

**Subagent 전략**

복잡한 작업을 하나의 AI에게 모두 맡기면 컨텍스트가 혼잡해지고 집중력이 떨어지므로 역할을 나누어 Subagent를 활용하는 방식이 좋습니다.

핵심은 AI가 하나의 작업에 집중하도록 역할을 분리하여 과도한 역할 부담을 줄이는 것이며, 메인 컨텍스트 윈도우를 깨끗하게 유지하고, 리서치·탐색·병렬 분석은 Subagent에게 위임하는 것입니다.

**Self-Improvement Loop**

사용자의 수정이 발생할 때마다 `tasks/lessons.md`에 패턴을 기록하여 같은 실수를 반복하지 않도록 하는 것이 좋습니다.

세션 시작 시 이 파일을 리뷰하게 하면 누적 학습 효과를 얻을 수 있습니다.

```jsx
# 핵심 원칙 요약
- Simplicity First : 모든 변경을 최대한 단순하게. 최소한의 코드만 수정
- No Laziness : 근본 원인을 찾는다. 임시 수정 금지
- Minimal Impact : 필요한 부분만 수정. 버그 유입 방지
- Verification Before Done : 동작 증명 없이 완료 선언 금지
- Demand Elegance : 중요한 변경은 "더 좋은 방법이 있는가?" 를 고민. 단순한 수정은 과도 설계 금지
- Autonomous Bug Fixing : 버그 리포트가 오면 질문하지 말고 바로 수정
```

**도구 자동화 - 훅과 검증**

린터, 포맷터, 검증 같은 반복 작업은 CLAUDE.md에 지시사항으로 넣기보다, 훅으로 자동 실행되게 만드는 편이 좋습니다.

> 훅 : 특정 이벤트가 발생했을 때 자동으로 실행되는 작업
> 

동작 흐름 : Claude가 파일 수정 → Hook 자동 감지 → 설정된 명령어 실행 → 결과를 Claude에게 피드백 → 문제가 있으면 Claude가 자동 수정

**검증 계층**

핵심 비즈니스 로직에 대한 단위 테스트가 가장 가치가 높으며, 테스트가 있으면 Claude가 코드를 수정한 뒤 자체적인 피드백 루프가 형성되어 결과 품질이 크게 향상됩니다.

### Claude.md 작성

---

```jsx
# Integrated Queueing System // 프로젝트 한 줄 소개

## 응답 언어 지침
## 프로젝트 개요         // 인프라 요약 포함, 한글 통일
## 빌드 및 실행
## 기술 스택          
## 아키텍처 핵심         // 트래픽 흐름 + Redis 키 구조 + 분산 안전성
## Gotchas              // 신규
## 코딩 컨벤션           // 신규
## Compact Instructions // 신규
```

CLAUDE.md에 모든 것을 넣는 대신, `.claude/rules/` 에 경로별 규칙을 분리했습니다.

**Gotchas 섹션**

Claude가 코드를 수정할 때 반복적으로 실수할 수 있는 함정을 모아둔 섹션입니다.

**코딩 컨벤션**

Claude Code가 코드를 생성할 때 지켜야 할 프로젝트 코딩 규칙

**Compact Instructions**

Claude Code는 대화가 길어지면 컨텍스트 윈도우를 절약하기 위해 이전 대화를 압축하는데, 그때 절대 빠뜨리면 안 되는 정보를 지정

### TIP

---

- `CLAUDE.md`는 팀 협업을 위한 파일이므로 기본적으로 Git에 커밋하여 팀원 간 공유하는 것이 권장됩니다.
- Claude가 잘못된 결과를 생성한 경우, 같은 실수를 반복하지 않도록 `CLAUDE.md`에 해당 instruction을 기록해 두는 것이 좋습니다.
- 반복적으로 수행하는 작업은 커맨드 형태로 저장하여, 같은 프롬프트를 매번 다시 입력하지 않도록 하는 것이 좋습니다.
    
    ex ) `add → commit → push`처럼 자주 반복되는 Git 작업 흐름은 하나의 명령으로 묶어 재사용하자
